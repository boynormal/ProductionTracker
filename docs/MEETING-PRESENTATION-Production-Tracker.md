# สรุประบบ Production Tracker — เอกสารประกอบการประชุม

เอกสารนี้สรุปข้อมูลพื้นฐาน หน้าจอ ขั้นตอนการใช้งาน และบทบาทตามสิทธิ์ผู้ใช้ สำหรับระบบติดตามการผลิตรายชั่วโมง (Next.js + PostgreSQL / Prisma)

---

## 1. ภาพรวมระบบ

| รายการ | รายละเอียด |
|--------|-------------|
| **วัตถุประสงค์** | บันทึกผลผลิตรายชั่วโมงต่อเครื่อง ติดตาม Breakdown / NG / เปลี่ยนรุ่น และใช้เป็นฐานรายงาน KPI |
| **เทคโนโลยีหลัก** | เว็บแอป (Next.js), ฐานข้อมูล PostgreSQL, เข้าสู่ระบบด้วยรหัสพนักงาน + รหัสผ่าน หรือ PIN ผ่าน QR |
| **กะการทำงาน** | กะ **DAY** (08:00–17:00) และ **NIGHT** (20:00–05:00) พร้อม OT ได้สูงสุด **11 ชั่วโมง/กะ** (ชั่วโมงที่ 9–11 เป็น OT) |
| **ชั่วโมงบันทึก (hour slot)** | ลำดับ **1–11** (ชั่วโมงที่ 1 = ชั่วโมงแรกของกะ) |

## 2. โครงสร้างเมนูหลัก (Sidebar)

| กลุ่ม | หน้า / เส้นทาง | หน้าที่โดยสรุป |
|--------|----------------|----------------|
| **หลัก** | `/` Dashboard | ภาพรวมวันนี้ เซสชันที่กำลังผลิต แจ้งเตือนที่ยังไม่อ่าน |
| | `/alerts` การแจ้งเตือน | รายการแจ้งเตือนระบบ (เช่น ชั่วโมงขาดการบันทึก) |
| **การผลิต** | `/production/record` บันทึกผลิต | กรอกข้อมูลรายชั่วโมงหลักของระบบ |
| | `/production/history` ประวัติ | ดูเซสชันและรายการรายชั่วโมงย้อนหลัง (ตามวันที่เริ่มต้น) |
| | `/production/report` รายงาน | สรุป / กรองตามสาย / เครื่อง / ช่วงวันที่ |
| | `/production/mtbf` MTBF/MTTR | วิเคราะห์ความพร้อมใช้งานและเวลาซ่อมเฉลี่ย |
| **ข้อมูลหลัก (Master)** | `/master/lines` สายการผลิต | จัดการสาย (Line) |
| | `/master/machines` เครื่องจักร | รายการเครื่อง — คลิกเข้า `/master/machines/[id]` ดูรายละเอียด รูป PM เป้าหมาย Part |
| | `/master/machines/qr` QR Code | สร้าง/พิมพ์ QR สำหรับเครื่องหรือสาย |
| | `/master/parts` Part | ข้อมูลชิ้นงาน / ลูกค้า |
| | `/master/problems` หมวดปัญหา | หมวด Breakdown / NG สำหรับเลือกตอนบันทึก |
| | `/master/departments` โครงสร้างองค์กร | แผนก → ฝ่าย → แผนกย่อย (สำหรับดู/อ้างอิงโครงสร้าง) |
| **ผู้ดูแลระบบ** *(มองเห็นเมนูเฉพาะบาง Role — ดูข้อ 5)* | `/admin/users` ผู้ใช้ | จัดการบัญชีผู้ใช้ (จำกัดสิทธิ์) |
| | `/admin/holidays` วันหยุด | กำหนดวันหยุด (มีผลต่อการแจ้งเตือนอัตโนมัติ) |
| | `/admin/notifications` การแจ้งเตือน (แอดมิน) | ดูประวัติ/จัดการแจ้งเตือนระดับระบบ |

## 3. การเข้าใช้งานแบบต่างๆ

### 3.1 เข้าสู่ระบบปกติ (พนักงานที่มี User ในระบบ)

1. เปิด `/login`
2. กรอก **รหัสพนักงาน (Employee Code)** และ **รหัสผ่าน**
3. หลังเข้าได้ จะเห็นเมนูตามสิทธิ์ Role

### 3.2 สแกน QR ที่เครื่อง (`/scan/[machineId]`)

1. สแกน QR → ระบบแสดงข้อมูลเครื่อง / สาย
2. กรอก **รหัสพนักงาน** (PIN) เพื่อยืนยันตัวตน
3. ไปขั้นตอนเลือก Part และบันทึกผลผลิต (ไม่บังคับล็อกอินแบบรหัสผ่านก่อน)

### 3.3 QR ระดับสาย (บันทึกแบบล็อกสาย)

- URL รูปแบบ **`/production/record?lineId=...`**
- เปิดหน้าได้ก่อน — ต้องกรอกรหัสพนักงานในหน้าเพื่อโหลดรายการเครื่องในสายนั้น
- หลังยืนยันแล้ว ใช้บันทึกได้เหมือนหน้าบันทึกปกติ (มี cookie สำหรับผู้ปฏิบัติงานจาก QR)

## 4. ขั้นตอนการคีย์บันทึกข้อมูลผลิตรายชั่วโมง (`/production/record`)

ลำดับทั่วไปที่ผู้ปฏิบัติงานใช้:

1. **เลือกเครื่องจักร** (ถ้าไม่ได้ล็อกมาจาก QR/สายแล้ว)
2. **เลือก Part** — ระบบดึง **เป้าหมายต่อชั่วโมง** จาก `machine_part_targets` ตามที่ตั้งค่าไว้ (ผู้ใช้ไม่ต้องคีย์ Target เอง)
3. **เลือกชั่วโมง (hour slot 1–11)** ให้ตรงกับช่วงเวลาจริง / กะ
4. **เลือกผู้ลงชื่อบันทึก (ผู้รับผิดชอบรายการ)** — ต้องระบุให้ครบตามที่ฟอร์มกำหนด
5. กรอก **จำนวน OK** และหมายเหตุ (ถ้ามี)
6. หากมีเหตุขัดข้อง:
   - เปิด **Breakdown** → ระบุเวลาเริ่ม/จบ นาทีหยุด และหมวดสาเหตุ
   - เปิด **NG** → ระบุจำนวน NG และหมวดสาเหตุ
7. **บันทึก** → ข้อมูลส่งไป API สร้าง/อัปเดต `hourly_records` ภายใต้ `production_sessions` ของวันและกะนั้น

**ข้อควรรู้ทางธุรกิจ**

- แต่ละชั่วโมงต่อเซสชัน **บันทึกได้หนึ่งรายการหลัก** — ถ้าชนกับ Part เดิม/ชั่วโมงเดิม ระบบจะแจ้งให้เปลี่ยน Part หรือชั่วโมง
- การแก้ไขรายการหลังบันทึกขึ้นอยู่กับ **สิทธิ์ Role** (ดูข้อ 5)

## 5. บทบาท (Role) และสิทธิ์สำคัญ

| Role | บทบาทโดยย่อ | สิ่งที่ทำได้ในระบบ (สรุปจากการออกแบบ) |
|------|--------------|----------------------------------------|
| **OPERATOR** | ผู้ปฏิบัติงานหน้าเครื่อง | สแกน QR, บันทึกรายชั่วโมง, ดู Dashboard/ประวัติ/รายงานตามที่เปิดสิทธิ์ |
| **SUPERVISOR** | หัวหน้ากะ / ผู้ควบคุม | บันทึกได้ + **แก้ไข**รายการผลิตที่บันทึกแล้ว (ร่วมกับบทบาทอื่นที่กำหนด), เข้า `/admin/notifications` ได้ |
| **ENGINEER** | วิศวกร / วางแผนเทคนิค | แก้ไข **ข้อมูลหลัก** (สาย, เครื่อง, Part, เป้าหมาย, หมวดปัญหา, รูปเครื่อง), แก้ไขรายการผลิต, ดู MTBF/MTTR |
| **MANAGER** | ผู้จัดการ | มองภาพรวม, รายงาน, ส่วน Admin บางส่วน (เช่น วันหยุด — ตามหน้า), จัดการเซสชันระดับสูงผ่าน API ที่กำหนด |
| **ADMIN** | ผู้ดูแลระบบ | จัดการ **ผู้ใช้**, ข้อมูลหลักครบวงจร, วันหยุด, การตั้งค่าระดับระบบที่เกี่ยวข้อง |

**หมายเหตุสำคัญ**

- เมนู **ผู้ดูแลระบบ** ใน Sidebar แสดงให้ **ADMIN และ MANAGER** แต่แต่ละหน้ามีข้อจำกัดต่างกัน เช่น หน้า **จัดการผู้ใช้** เปิดได้เฉพาะ **ADMIN** เท่านั้น
- การ **แก้ไข/ลบ** ข้อมูลการผลิตหลังบันทึก: ใช้สิทธิ์ระดับ **SUPERVISOR, ENGINEER, MANAGER, ADMIN** (ตาม API)

## 6. การดูรายงานและ KPI

### 6.1 Dashboard (`/`)

- สรุปสถานะวันนี้ เซสชัน **IN_PROGRESS** จำนวนเครื่องที่เปิดงาน และแจ้งเตือนที่ยังไม่อ่าน

### 6.2 ประวัติ (`/production/history`)

- ดูรายการเซสชันและรายละเอียดรายชั่วโมง (OK, Breakdown, NG) ย้อนหลัง

### 6.3 รายงาน (`/production/report`)

- กรองตามสาย / เครื่อง / ช่วงวันที่ — ใช้สรุปผลการผลิตและเปรียบเทียบกับเป้า

### 6.4 MTBF / MTTR (`/production/mtbf`)

- **MTBF** (Mean Time Between Failures): อิงเวลาทำงานรวมและจำนวนครั้งที่เสียเมื่อมี breakdown ปิดงานแล้ว
- **MTTR** (Mean Time To Repair): เฉลี่ยเวลาซ่อมจาก breakdown ที่จบแล้ว  
- การคำนวณในระบบใช้เซสชันที่ **สถานะ COMPLETED** และ breakdown ที่มี **เวลาสิ้นสุด** ตามนิยามในโค้ด

### 6.5 OEE (โดยสรุป)

จากข้อมูลรายชั่วโมงและ downtime:

- **Availability** — เวลาเทียบกับ downtime
- **Performance** — ผลผลิตจริงเทียบเป้า
- **Quality** — OK เทียบกับ OK+NG  

คูณกันเป็น **OEE**

## 7. การแจ้งเตือนอัตโนมัติ

- ระบบตรวจ (Cron / API `notifications/check`) เป็นระยะว่าในช่วงกะปัจจุบันมี **ชั่วโมงที่ควรบันทึกแต่ยังไม่มีข้อมูล** หรือไม่
- **ไม่แจ้ง**ในวันหยุดที่ตั้งใน `/admin/holidays`
- สามารถส่งแจ้งเตือนไปยัง Telegram (ถ้าตั้งค่า Bot ในระบบ)

---

## 8. หน้าที่ดูแลข้อมูลตามประเภทงาน (แนวทางองค์กร)

ระบบไม่ได้ผูก “แผนก” กับเมนูแก้ไขแบบ 1:1 ในโค้ดทุกจุด — ใช้ **Role + ความรับผิดชอบจริง** เป็นหลัก โดยแนะนำดังนี้:

| ประเภทข้อมูล | แนะนำให้แผนก / ตำแหน่งที่เกี่ยวข้อง |
|--------------|-------------------------------------|
| **บันทึกผลิตรายชั่วโมง, ความถูกต้องของ OK/NG/Breakdown** | แผนกผลิต / หัวหน้ากะ (OPERATOR บันทึก — SUPERVISOR ตรวจและแก้ไขเมื่อจำเป็น) |
| **สายการผลิต, เครื่อง, เป้าหมาย Part ต่อเครื่อง, รูป PM** | วิศวกรกระบวนการ / แผนกวางแผนการผลิต / Maintenance ตามนโยบายโรงงาน (ENGINEER / ADMIN) |
| **Part, ลูกค้า, หมวดปัญหา** | แผนกวางแผน / QC / วิศวกร (ENGINEER / ADMIN) |
| **ผู้ใช้, สิทธิ์, โครงสร้างผู้ใช้กับ Part** | HR / IT / ผู้ดูแลระบบ (หน้าผู้ใช้: **ADMIN** เท่านั้น) |
| **วันหยุด, การตั้งค่าแจ้งเตือน** | ฝ่ายบริหารหรือแผนกที่ได้รับมอบหมาย (**ADMIN / MANAGER** ตามหน้า) |
| **รายงานภาพรวม, KPI** | ผู้จัดการสาย / ผู้จัดการโรงงาน (**MANAGER**) |

## 9. หัวข้อสำหรับนำเสนอ (สไลด์สั้นๆ)

1. **ปัญหาเดิม** — ข้อมูลกระจาย ไม่ทันต่อการตัดสินใจรายชั่วโมง  
2. **ทางออก** — ระบบเดียว: บันทึกจาก QR หรือหน้าเว็บ, เก็บเป็นฐานข้อมูลกลาง  
3. **โฟลว์หลัก** — Login หรือ PIN → เลือกเครื่อง/Part → ชั่วโมง → OK / Breakdown / NG → บันทึก  
4. **รายงาน** — ประวัติ, รายงานสรุป, MTBF/MTTR, Dashboard  
5. **สิทธิ์** — แยก OPERATOR ถึง ADMIN; แก้ Master และผู้ใช้ตาม Role  
6. **ขั้นต่อไป** — การฝึกอบรมจุดบันทึก, การกำหนดผู้รับผิดชอบ Master Data ต่อสาย
